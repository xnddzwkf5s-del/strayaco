Configuring Multiple API Providers in OpenClaw

OpenClaw DeepSeek Anthropic Ollama Configuration Security Cost Optimisation

A complete guide to adding DeepSeek, Anthropic, Ollama, and OpenAI-compatible API providers to OpenClaw — with provider comparison, cost analysis, model routing, and security hardening.

Overview

OpenClaw doesn't ship with hardcoded provider options during installation. Instead, it asks you to select an API format — and DeepSeek is not listed as a named provider. The correct choice during setup is "OpenAI Compatible", which opens up any service using the OpenAI chat completions format, including DeepSeek, Together AI, Groq, Perplexity, OpenRouter, and local Ollama instances.

This guide covers configuring multiple providers in the same openclaw.json, routing tasks to the right model, and hardening your config against leaks.

Why Multiple Providers?

Running one API provider is simple. Running multiple gives you:

  • Fallback resilience — if DeepSeek is down, the agent falls back to Anthropic, Ollama, or another provider
  • Cost optimisation — DeepSeek Chat costs ~$0.50/day for typical agent use; Claude Sonnet can burn $50/day on the same workload. Route simple tasks to cheap models, programming to premium models.
  • Privacy segmentation — sensitive tasks (email processing, document analysis) route through local Ollama models; public tasks (web search, blog writing) use cloud APIs
  • Model specialisation — different models excel at different things; DeepSeek for code, Claude for writing, local Qwen for fast structured tasks

The configuration shown here uses four providers simultaneously: DeepSeek (primary), Anthropic (premium fallback), Ollama (local privacy), and an OpenAI placeholder for future use.

The "OpenAI Compatible" Setup

During OpenClaw's initial setup wizard, you'll see a list of provider options:

Which LLM provider do you want to use?
> OpenAI
> Anthropic
> Ollama
> OpenAI Compatible

Select "OpenAI Compatible". This does not mean you're using OpenAI — it means the provider speaks the OpenAI chat completions protocol, which is the de facto standard for API-based LLMs. DeepSeek, Together AI, Groq, Perplexity, OpenRouter, and dozens of others all use this format.

After selecting this option, the wizard asks for:

  1. Base URL — the API endpoint (e.g. https://api.deepseek.com/v1)
  2. API Key — your authentication key
  3. Model ID — the model name (e.g. deepseek-chat)

This creates a provider entry in openclaw.json. You can then edit the config file directly to add additional providers.

Provider Configuration Walkthrough

Provider 1: DeepSeek (Primary)

DeepSeek's API uses OpenAI-compatible format. There are two relevant models:

deepseek-chat — the flagship model. Fast, excellent at code and structured tasks, cost-effective.
deepseek-reasoner — the reasoning model with extended chain-of-thought. Slower but better at complex logic.

{
  "custom-api-deepseek-com": {
    "baseUrl": "https://api.deepseek.com/v1",
    "apiKey": "sk-your-deepseek-api-key",
    "api": "openai-completions",
    "models": [
      {
        "id": "deepseek-chat",
        "name": "deepseek-chat (Custom Provider)",
        "reasoning": false,
        "input": ["text"],
        "contextWindow": 128000,
        "maxTokens": 8192
      },
      {
        "id": "deepseek-reasoner",
        "name": "deepseek-reasoner (Custom Provider)",
        "reasoning": false,
        "input": ["text"],
        "contextWindow": 128000,
        "maxTokens": 64000
      }
    ]
  }
}

Key details:

  • baseUrl includes /v1 — the API version path. Without this, requests fail silently
  • api: "openai-completions" — tells OpenClaw which protocol to speak
  • DeepSeek supports up to 128K context and generous rate limits
  • The provider key (custom-api-deepseek-com) is an arbitrary identifier but should be descriptive

Provider 2: Anthropic (Premium Fallback)

Anthropic uses its own message format, not OpenAI-compatible. This requires a separate provider type:

{
  "anthropic": {
    "baseUrl": "https://api.anthropic.com/v1",
    "apiKey": "sk-ant-your-anthropic-api-key",
    "api": "anthropic-messages",
    "models": [
      {
        "id": "claude-sonnet-4-6",
        "name": "Claude Sonnet 4.6",
        "api": "anthropic-messages",
        "contextWindow": 200000,
        "maxTokens": 8192
      },
      {
        "id": "claude-3-5-haiku-20241022",
        "name": "Claude 3.5 Haiku",
        "contextWindow": 200000,
        "maxTokens": 8192
      }
    ]
  }
}

Key differences from OpenAI-compatible:

  • api: "anthropic-messages" — completely different request/response format
  • Each model must explicitly declare "api": "anthropic-messages" (some models inherit from the provider, but explicit is safer)
  • Anthropic's API key starts with sk-ant-
  • Claude models have very large context windows (200K tokens)

Provider 3: Ollama (Local)

Ollama runs locally on your machine and serves an OpenAI-compatible endpoint:

{
  "ollama": {
    "baseUrl": "http://localhost:11434/v1",
    "apiKey": "ollama-local",
    "api": "openai-completions",
    "models": [
      {
        "id": "qwen3.5:latest",
        "name": "Qwen 3.5 (latest)",
        "contextWindow": 128000,
        "maxTokens": 4096
      },
      {
        "id": "deepseek-r1:7b",
        "name": "DeepSeek R1 7B",
        "contextWindow": 128000,
        "maxTokens": 8192
      }
    ]
  }
}

Key details:

  • The apiKey is arbitrary — Ollama's local server doesn't validate it, but OpenClaw requires a non-empty value
  • baseUrl points to localhost:11434 (Ollama's default port)
  • Local models have smaller practical context windows despite what they advertise
  • Models need to be pulled first via ollama pull modelname

Provider 4: Placeholder / Unused Provider

You can keep an empty provider for future use:

{
  "openai": {
    "baseUrl": "http://localhost:8000/v1",
    "apiKey": "placeholder",
    "api": "openai-completions",
    "models": []
  }
}

This has no impact on performance. An empty model list means OpenClaw ignores it when resolving model references.

The Default Model Configuration

After adding providers, you need to tell OpenClaw which model(s) to use by default and how to fall back:

{
  "agents": {
    "defaults": {
      "model": {
        "primary": "custom-api-deepseek-com/deepseek-chat",
        "fallbacks": [
          "custom-api-deepseek-com/deepseek-reasoner",
          "anthropic/claude-sonnet-4-6",
          "ollama/deepseek-r1:7b"
        ]
      },
      "models": {
        "custom-api-deepseek-com/deepseek-chat": {
          "alias": "deepseek-chat",
          "params": {
            "max_tokens": 8192,
            "timeout": 60000,
            "max_retries": 3
          }
        },
        "anthropic/claude-sonnet-4-6": {
          "alias": "claude-sonnet-4-6"
        },
        "ollama/qwen3.5:latest": {},
        "ollama/deepseek-r1:7b": {}
      }
    }
  }
}

Provider and model references use the format: provider-id/model-id

The alias field lets you override a model's display name in the UI and when selecting models per-session.

Fallback order is critical. OpenClaw tries the primary model first. If it fails (timeout, rate limit, 503), it tries the first fallback, then the second, and so on. This means:

  1. DeepSeek chat (fast, cheap)
  2. DeepSeek reasoner (slower, more thorough)
  3. Claude Sonnet (premium, very capable)
  4. Local DeepSeek R1 (slow, private, last resort)

How OpenClaw Routes Models

The model routing system uses dot-notation paths: provider/model.

# Use the default model (as configured in agents.defaults.model.primary)
/deepseek-chat

# Switch to a specific provider and model
/anthropic/claude-sonnet-4-6

# Use a local model
/ollama/qwen3.5:latest

When you reference a model in a session, OpenClaw looks it up in the default model.primary first, then checks fallbacks if the primary fails. Each session can override the model selection independently.

The models section under agents.defaults is separate from the providers section. Providers define what's available. The models section defines how each one is configured for your agent — timeouts, retries, token limits, aliases.

The Real Cost Difference: Why Model Selection Matters

This is not theoretical. Running Claude Sonnet as your only model will cost you real money. Running DeepSeek Chat for the same workload will cost you almost nothing.

The numbers from real usage:

ScenarioDaily CostMonthly CostAnnual Cost
Anthropic Claude Sonnet (default)~$50/day~$1,500~$18,000
DeepSeek Chat + Reasoner (default)~$0.50/day~$15~$180
Hybrid: DeepSeek default, Claude for programming~$5-10/day~$150-300~$1,800-3,600

That's not a typo. Claude Sonnet can burn through $50 per day of heavy agent usage. DeepSeek Chat handles the same volume for pocket change.

Why is the difference this extreme?

  • Pricing per million tokens: DeepSeek Chat is ~$0.14 input / $0.28 output. Claude Sonnet is ~$3.00 input / $15.00 output. Sonnet is roughly 20-50x more expensive depending on the input/output ratio.
  • Agent workloads are token-heavy. An agent doesn't just send a prompt and get a reply. It sends the full conversation history, tool call results, system prompts, and context — every time. A single agent session can consume 50,000-200,000 input tokens per turn.
  • Cron jobs multiply the cost. Six cron jobs running daily means 6x the token consumption. A single Sonnet-powered cron job handling email processing can cost $3-5 per run.

The hybrid strategy:

DeepSeek Chat handles 90% of agent operations — email triage, weather checks, backup verification, blog formatting, file management, web searches. These are structured, predictable tasks where DeepSeek performs perfectly well.

Claude Sonnet is reserved for the 10% where it genuinely matters — complex programming tasks, nuanced writing, multi-step reasoning where you can't afford errors.

# Default session: DeepSeek Chat (cheap, fast, good enough)
/deepseek-chat

# Programming session: Claude Sonnet (premium, best for code)
/anthropic/claude-sonnet-4-6

This brings daily cost down from $50 to $5-10. The programming work still gets the best model. Everything else runs on the efficient workhorse.

What about the fallback chain?

Even with fallback configured, Claude should never be reached in normal operation because DeepSeek has excellent uptime. The fallback is insurance for the rare case DeepSeek's API is down — it means the agent keeps working at premium cost rather than failing entirely.

Local models for true zero cost:

For fully private, offline-capable tasks, Ollama local models cost nothing. A weekly email processing run that would cost $0.05 on DeepSeek or $3.00 on Sonnet costs $0.00 on a local 7B model. Speed and quality trade-offs apply.

Security Hardening

Your openclaw.json contains API keys for multiple services. A leak means someone can drain all of them.

1. Use Environment Variables (Recommended)

Replace hardcoded API keys with environment variable references:

{
  "custom-api-deepseek-com": {
    "apiKey": "${DEEPSEEK_API_KEY}"
  },
  "anthropic": {
    "apiKey": "${ANTHROPIC_API_KEY}"
  },
  "skills": {
    "entries": {
      "agentmail": {
        "apiKey": "${AGENTMAIL_API_KEY}"
      }
    }
  }
}

Then set these in your shell profile (~/.zshrc) or a .env file:

export DEEPSEEK_API_KEY="sk-your-key"
export ANTHROPIC_API_KEY="sk-ant-your-key"
export AGENTMAIL_API_KEY="am_us_your-key"

OpenClaw resolves ${VAR} placeholders at startup. If a variable is unset, the config will fail to load — which is safer than falling back to a default.

2. Lock Down File Permissions

chmod 600 ~/.openclaw/openclaw.json
chmod 700 ~/.openclaw

This prevents other users on the system (or any process running as a different user) from reading your config.

3. Exclude Config from Backups

If you back up your workspace publicly (e.g., website backups pushed to GitHub), make sure OpenClaw config is excluded:

# .gitignore or backup exclusion script
openclaw.json

The config lives at ~/.openclaw/, which should be outside any public backup scope anyway. But verify.

4. Set Rate Limits on the Gateway

Protect against brute force of the control panel:

{
  "gateway": {
    "auth": {
      "rateLimit": {
        "maxAttempts": 10,
        "windowMs": 60000,
        "lockoutMs": 300000
      }
    }
  }
}

This locks out the IP for 5 minutes after 10 failed auth attempts within 60 seconds.

5. Keep Unused Providers Empty

If you're not using OpenAI directly, keep its model list empty ([]). An unused provider with models configured could theoretically be selected accidentally or by a confused agent.

6. Trusted Proxies for Network Security

If your agent is accessible over a local network, specify which IPs are trusted:

{
  "gateway": {
    "trustedProxies": [
      "127.0.0.1",
      "::1",
      "192.168.1.0/24"
    ]
  }
}

This prevents external hosts from spoofing trusted IP headers.

Troubleshooting

"Failed to resolve model" error

Cause: The provider key or model ID doesn't match. References use the format provider-name/model-id.

Fix: Verify the provider key in the providers section matches what you use in agents.defaults.model.fallbacks.

Provider key: "custom-api-deepseek-com"
Reference:     "deepseek/deepseek-chat"  → Wrong
Reference:     "custom-api-deepseek-com/deepseek-chat"  → Correct

"Connection refused" for Ollama

Cause: Ollama isn't running or isn't listening on the expected port.

# Check if Ollama is running
ollama list
# Start Ollama
ollama serve

"401 Unauthorized" for DeepSeek

Cause: Invalid API key or the key hasn't been activated yet. Verify at api.deepseek.com.

Model responses are truncated

Cause: maxTokens is set too low in the model configuration.

{
  "params": {
    "max_tokens": 16384
  }
}
