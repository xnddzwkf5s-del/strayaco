AI-Powered Website Publishing Workflow

GitHub Pages Cloudflare Git Versioning AI Agent DevOps

How we build, version, back up, and publish a static website using GitHub Pages, Cloudflare Workers, and an AI assistant — with real examples from the Straya Co project.

Overview

Publishing a website used to mean FTPing files to a server or clicking "Deploy" in a CMS. Modern static site publishing is simpler, but it introduces a different set of decisions: where to host, how to version, how to back up, and how to automate the pipeline.

This post walks through the full publishing workflow for Straya Co, a static website running on GitHub Pages with Cloudflare Worker proxying, managed through Git versioning and assisted by an AI agent. It covers the actual tools, commands, and lessons learned from dozens of deployments.

The stack is intentionally simple. No build tools, no frameworks, no databases. Just HTML files pushed to a GitHub repo, proxied through Cloudflare for the custom domain and HTTPS.

The Publishing Workflow

Step 1: Make Changes Locally

All work happens in a local repo:

cd /path/to/repo
python3 -m http.server 4000

Serving locally at http://localhost:4000 lets you preview changes before they go live. The site is plain HTML — no build step, no compilation, no dependencies.

Step 2: Back Up

Before every publish, a timestamped zip archive of the entire repo (excluding .git) is created:

cd /path/to/projects
zip -r "backup_$(date +%Y%m%d_%H%M%S).zip" repo/ --exclude "repo/.git/*"

This produces files like straya_backup_20260429_220456.zip. It's a safety net — if something goes wrong during publishing, you have a full snapshot from immediately before the commit.

Step 3: Commit and Tag

git add -A
git commit -m "v4.9 - Add blog: Configuring Multiple API Providers"
git tag v4.9

Version tags are sequential (v4.0, v4.1, v4.2...). Each blog post or feature add gets a version bump. The commit message includes the version number and a one-line description.

Step 4: Push Both Branch and Tag

git push origin main
git push origin v4.9

Two pushes are needed — one for the branch (which triggers GitHub Pages rebuild) and one for the tag (for version tracking).

Step 5: Cloudflare Cache Delay

GitHub Pages deploys in seconds, but Cloudflare's edge cache takes a moment. After pushing, there's a brief delay (30–60 seconds) before the site reflects changes at the custom domain. If you need instant updates, you can purge the Cloudflare cache manually.

Automated Versioning

Versioning is manual but disciplined. The approach:

  1. Tags are the source of truth. Each published version has a corresponding Git tag.
  2. Increment always. Every deploy gets a new version number. No overwriting, no forgotten tags.
  3. Descriptive messages. v4.4 - Add blog: Connecting Telegram and Email to OpenClaw tells you what changed without looking at the diff.
  4. No version file needed. The Git tag is the version. The site itself doesn't display a version number — it's a development tool, not a user-facing feature.

The version history tells a story. Looking at git tag output:

v4.9 - Add blog: Configuring Multiple API Providers
v4.8 - Add blog: AI-Powered Cron Jobs
v4.7 - Add blog: Email Processing with an AI Agent
v4.6 - Add blog: AI-Powered Website Publishing Workflow
v4.5 - Add blog: Watching YouTube Videos
v4.4 - Add blog: Connecting Telegram and Email
v4.3 - Add blog landing page + first post
v4.2 - Add openclaw.html test page
v4.1 - Repo migration to new GitHub account
v4.0 - Initial push to new repo

This is more useful than commit hashes. You can jump to any published state with git checkout v4.3 and see exactly what was live at that point.

The AI Agent's Role

The OpenClaw AI agent handles most of the publishing workflow automatically:

  1. Creates the content — writes blog drafts based on the user's setup and preferences
  2. Sanitises sensitive information — replaces real tokens, emails, and domains with placeholders before publication
  3. Generates the HTML — converts markdown drafts into styled pages matching the site's design
  4. Runs the backup — zips the repo with a timestamp filename
  5. Commits, tags, and pushes — builds the git commands and executes them
  6. Verifies the deployment — checks that the live page returns the expected content

All of this happens from a single instruction: "Publish the YouTube blog post." The agent knows the workflow from memory and executes the steps in sequence.

Lessons Learned

Lesson 1: Backups Before Publishing, Always

Early in the project, I published without a backup and accidentally pushed a broken commit. Recovering meant digging through git reflog. Now the backup step is non-negotiable — and it's the first thing the agent does.

The zip backup is intentionally separate from Git. If the remote repo becomes corrupted, or if you need to recover files that were .gitignored, the zip has everything.

Lesson 2: Git Tags Are Better Than Version Numbers in Files

Some projects put a version number in a version.txt or in the HTML footer. This works, but it's one more thing to update and it pollutes your diff. Git tags are metadata — they live alongside the code without modifying it.

The trade-off is that tags aren't pushed automatically with git push. You have to explicitly push them with git push origin vX.X. Forgetting this is a common mistake.

Lesson 3: The Cloudflare Worker Adds Latency and Complexity

The Cloudflare Worker sits between GitHub Pages and the user. It provides the custom domain, handles HTTPS at the edge, and allows URL rewriting. But it also adds a deployment step, causes a cache delay after publishing, and introduces a failure mode (if the Worker errors, the site is down even if GitHub Pages is healthy).

For a simple static site, GitHub Pages + a CNAME record would be simpler. The Worker is only necessary if you need advanced routing or multiple backends under the same domain.

Lesson 4: Token Auth in Local Git Remotes Is Fragile

Early on, the remote URL included a personal access token embedded in the URL. This works, but the token is exposed in .git/config (plaintext), shell history, and process listings during pushes. The current workflow resets the remote URL after each push to avoid leaving the token in config. A cleaner solution would be using SSH keys or the git credential helper.

Lesson 5: AI Agents Speed Up Repetitive Work

The publishing workflow involves seven distinct steps (backup, commit, tag, push branch, push tag, reset remote, verify). Doing this manually takes about two minutes and is error-prone. The agent does it in about 15 seconds with zero mistakes.

The trade-off is that the agent needs detailed context about the workflow — it can't infer "my usual process" without being told. But once the process is documented in its memory, every subsequent publish is automatic.

What Could Be Better

CI/CD Pipeline

Currently, publishing requires a local agent to run git commands. A GitHub Action could automate this: on push to main, build and deploy automatically; on creating a tag, create a GitHub Release with release notes. This would eliminate the manual push step and the "forgot to push tags" problem.

Automated Backups to Cloud Storage

Local zip backups are fine, but they live on the same machine as the repo. Off-site backups (to S3, Backblaze B2, or Dropbox) would protect against drive failure.

A Changelog Page

The git tags contain all the release notes, but they're not exposed on the website. A /changelog page that lists versions with descriptions would turn the tag history into a user-facing feature.

Semantic Versioning

The current version scheme (v4.0, v4.1...) doesn't communicate whether a release is a major feature, a minor improvement, or a bug fix. Moving to proper semver (v4.0.0, v4.1.0, v4.1.1) would be more informative, but requires discipline around what constitutes a "major" vs "minor" change.

Final Thoughts

The Straya Co publishing workflow is deliberately simple. It proves that you don't need a complex CI/CD pipeline, a static site generator, or a CMS to publish a professional website. You need:

  1. A Git repo
  2. A hosting provider (GitHub Pages is free)
  3. A domain with DNS
  4. A backup strategy
  5. A repeatable process (automated or documented)

The AI agent makes the process fast and consistent, but the core workflow works just as well with manual commands. The key is having a process at all — knowing exactly what steps to take, in what order, every time you publish.
