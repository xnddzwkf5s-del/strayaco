Network Drive Mapping That Actually Works

A simple script that makes NAS drives behave reliably — no more "disconnected network drive" errors every time someone reboots.

NAS Network Drive PowerShell Windows File Sharing Small Business IT Automation

The Pain

If you've got a NAS in your office, you know the story. Everyone connects to it manually — or a login script that mostly works. Until someone reboots. Or switches from Wi-Fi to Ethernet. Or works from home and the drive stubbornly tries to connect.

Suddenly there's a red X on the drive icon, someone can't save their work, and IT gets a ticket.

The fix is a simple PowerShell script that checks the network first, maps the drives, and — most importantly — only runs when it should. No more failed drive letters at login. No more corrupted connections.

The Setup

The script lives in a dedicated folder structure:

C:\Users\Username\Documents\Scripts\NAS\
  ├── Config\
  │   └── NASConfig.json       # Your settings
  ├── Scripts\
  │   └── Map-Drives.ps1       # The script
  └── Logs\                    # Only created if logging is on
      └── nas_mapping.log

One JSON file controls everything, so you don't need to edit the script itself.

The Config

{
  "UserSettings": {
    "TargetUser": "YourUsername"
  },
  "Network": {
    "ValidationMethod": "Both",
    "AllowedSSIDs": [
      "Office-WiFi",
      "Guest-Network"
    ],
    "AllowedSubnets": [
      "192.168.86."
    ]
  },
  "NAS": {
    "ServerName": "192.168.1.100",
    "Username": "nas_username"
  },
  "Mappings": {
    "P": { "ShareName": "Projects", "Description": "Project Files" },
    "S": { "ShareName": "Shared", "Description": "Team Shared Drive" },
    "B": { "ShareName": "Backups", "Description": "Daily Backups" }
  }
}

Key bits:

  • Network validation — the script checks SSID and/or subnet before attempting to map. Working from home? It skips the mapping gracefully instead of hanging.
  • Drive letters — assign any letter (P:, S:, B:, etc.) and point it to the correct share on your NAS.
  • Credentials — stored once in Windows Credential Manager, so the script isn't handling plain text passwords.

How It Works

When the script runs:

  1. Check network — is the machine on the right SSID or subnet? No → exit cleanly.
  2. Validate NAS — can we reach the NAS? No → log the error, exit.
  3. Remove stale connections — clean up any broken drive letters from previous sessions.
  4. Map drives — connect P: to \\NAS\Projects, S: to \\NAS\Shared, etc.
  5. Log results — if logging is enabled, write success/failure to the log file.

Automating It

The script runs via Task Scheduler at user logon. Setup takes two minutes:

  1. Open Task Scheduler
  2. Create a Basic Task → trigger "At log on"
  3. Action: Start a program
  4. Program: powershell.exe
  5. Arguments: -NoProfile -ExecutionPolicy Bypass -File "C:\Users\Username\Documents\Scripts\NAS\Scripts\Map-Drives.ps1"

That's it. Every time someone logs in, the script runs, checks the network, and maps the drives.

Why This Beats a Manual Connection

A manually mapped drive (via "Map Network Drive" in File Explorer) will:

  • Stay connected until it breaks — then show a red X forever
  • Try to reconnect on every network change, sometimes causing delays
  • Fail silently if the NAS is unreachable at login

The script does the opposite:

  • Only maps when conditions are right
  • Cleans up stale connections first
  • Logs errors so you can see what happened
  • Exits cleanly if the network isn't appropriate (home, coffee shop, etc.)

When You'd Use This

Small business with a Synology, QNAP, or TrueNAS box. Multiple staff, multiple shared folders, all on Windows machines. You want reliable drive letters without enterprise tools like Group Policy.

Mixed environment — desk and laptop. People move between wired and wireless, or work from home occasionally. The network validation keeps stale drives from gumming up the works.

Office with no Active Directory. For a lot of small businesses, AD is overkill. This script gives you some of the same benefits without the infrastructure cost.

What About Security?

The script doesn't store passwords. Windows credentials are either saved once via Credential Manager or entered interactively on first connection. The config file holds the NAS address and share names — no secrets.

If you want to take it further, you can:

  • Restrict which machines can run the script
  • Enable auditing via the log file
  • Pair with a VPN check for remote access

The Bottom Line

Network drive mapping seems trivial until it breaks repeatedly. A 50-line PowerShell script with a JSON config eliminates the most common failure points and makes shared storage behave like it should: invisibly and reliably.
