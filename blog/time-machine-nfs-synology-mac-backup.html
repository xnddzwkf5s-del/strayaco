Future-Proofing Your Mac Backups: Moving from SMB to NFS with Synology NAS

Apple is phasing out SMB support for Time Machine. Here's what that means and how to move to NFS before it breaks.

Time Machine NFS SMB Synology NAS macOS Backups Data Management

The Headlines

Apple has quietly signalled that SMB protocol support for Time Machine backups will be deprecated in future macOS versions. If you're backing your Macs up to a Synology NAS over SMB — and a lot of people are — that's a ticking clock.

The fix is NFS (Network File System). It's more stable, better suited to network backups, and aligns with where Apple is heading. This guide walks through the migration, step by step.

Why Apple Is Making This Change

SMB was never really designed for Time Machine. It's a file-sharing protocol, not a backup protocol. Over the years, Apple has had to add workarounds and patches to make SMB work reliably with Time Machine's sparse bundle format — and those patches have been brittle.

NFS handles the locking, the large files, and the network overhead more naturally. Moving to NFS means fewer corrupted backups, fewer "backup disk not available" errors, and less friction when macOS updates break something downstream.

What This Means for You

If your backup setup looks like this:

Mac → network → Synology NAS → SMB share → Time Machine

Then you have a choice: migrate proactively, or wait until an macOS update forces your hand. Proactive is better. The migration isn't difficult, and once it's done, it just works.

What You'll Need

  • A Synology NAS (any model running DSM 7.x or later)
  • A Mac running macOS Ventura, Sonoma, or Sequoia
  • Admin access to both
  • A network connection between them (wired is better, Wi-Fi works)

Step 1 — Enable NFS on Your Synology

1. Open Control Panel → File Services → NFS

2. Tick Enable NFS and click Apply

3. Under NFS Permissions, set:

  • NFSv4 support: enabled (fallback to v3 if needed)
  • Transport: TCP (default)
Note: DSM 7.2+ defaults to NFSv4 with NFSv3 fallback. That's fine — macOS handles both. If you're on an older DSM, just make sure NFS is on.

Step 2 — Create a Dedicated Time Machine Share

1. Open Control Panel → Shared Folder → Create

2. Name it something obvious like timemachine

3. Click Next through the defaults, but on the NFS Permissions screen:

  • Hostname/IP: Enter your Mac's IP, or a subnet like 192.168.86.0/24 if you have multiple Macs
  • Privilege: Read/Write
  • Squash: No mapping
  • Security: sys
  • Async: Enable

4. Finish creation.

Pro tip: Use a subnet range rather than a single IP. That way you don't have to reconfigure when your Mac gets a new DHCP lease. Better yet, set a DHCP reservation on your router for each Mac's MAC address.

Step 3 — Mount the NFS Share on Your Mac

Open Terminal and run:

# Create a mount point
sudo mkdir -p /Volumes/timemachine

# Mount the NFS share (replace nas_ip with your Synology's IP)
sudo mount -t nfs -o resvport,locallocks nas_ip:/volume1/timemachine /Volumes/timemachine

If that works, you'll see the share in Finder under Locations. If it fails, double-check:

  • NFS is enabled on the Synology
  • The NFS permission rule covers your Mac's IP
  • Your Mac can reach the NAS (try ping nas_ip)

Step 4 — Point Time Machine at the NFS Volume

1. Open System Settings → General → Time Machine Backup

2. Click Add Backup Disk (or the + button)

3. Select the mounted NFS volume

4. Time Machine will ask if you want to erase it or use it as-is (if it's brand new, let it format it)

5. Done. Time Machine will start its first backup within 120 seconds.

One quirk: Time Machine may ask for authentication briefly during the first few backups. This is normal — it's establishing a secure connection to the NFS mount.

Making It Permanent

That mount command won't survive a reboot. To make it stick:

1. Open System Settings → General → Login Items

2. Click the + under Allow in the Background

3. Add Terminal (or create a small launch script)

Alternatively, add an entry to /etc/fstab:

echo "nas_ip:/volume1/timemachine /Volumes/timemachine nfs rw,resvport,locallocks,noatime 0 0" | sudo tee -a /etc/fstab

Or for a cleaner approach, create a LaunchDaemon that mounts it automatically. If this sounds fiddly, Straya Co can set it up for you in a few minutes.

What About Existing SMB Backups?

If you have old SMB-based Time Machine backups you want to keep, here's the pragmatic route:

  1. Start fresh with NFS for ongoing backups
  2. Keep the old SMB volume around as an archive (but unmounted)
  3. Once you're confident the NFS backups are working, you can delete the old sparse bundle

Time Machine will do a full initial backup to the new NFS volume, so you won't lose coverage.

A Note on Performance

NFS over a wired Gigabit connection is noticeably more reliable than SMB for Time Machine. Wireless will work, but expect slower initial backups (especially the first one, which is always the biggest).

If you're on Wi-Fi, consider:

  • A 5 GHz connection (not 2.4 GHz)
  • Scheduling backups during off-peak hours
  • Using macOS's built-in throttling (it handles this well)

The Bottom Line

Apple deprecating SMB for Time Machine isn't a crisis. It's an opportunity to move to a better protocol. NFS is more stable, more performant, and more aligned with where macOS is going. The migration takes about 15 minutes of hands-on work, and once it's done, you can forget about it.
