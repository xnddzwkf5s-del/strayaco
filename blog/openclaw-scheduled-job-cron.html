AI-Powered Cron Jobs: Automating Your Life with OpenClaw

OpenClaw Cron Automation Scheduling AI Agent Workflow

How to use OpenClaw's scheduling system to automate everything from weather alerts to config backups to email processing — with real cron configurations and lessons learned.

Overview

Cron jobs have been around since 1975. They're simple: "run this command at this time." But with an AI agent, cron jobs become something more powerful: "run this task with intelligence."

Instead of a shell script that sends a static message, an agent-powered cron job can:

  • Check conditions and decide whether to act
  • Summarise and contextualise information
  • Route messages to the right channel
  • Handle errors gracefully
  • Execute multi-step workflows

This post covers the real-world cron setup powering an OpenClaw AI agent — handling weather alerts, email processing, and configuration backups.

The Jobs at a Glance

Job 1: The Intelligent Weather Alert

Schedule: Every Friday at 6 PM

{
  "name": "Friday Weather Check",
  "schedule": {
    "kind": "cron",
    "expr": "0 18 * * 5",
    "tz": "Australia/Melbourne"
  },
  "target": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Check weather for our location for tomorrow (Saturday). Use Open-Meteo API to get forecast data. If rain is predicted (>1mm), send a brief alert. Otherwise, do nothing."
  },
  "delivery": {
    "mode": "none"
  }
}

This is a perfect example of the "intelligent" cron pattern. Instead of sending a weather report every Friday whether you need it or not, the agent:

  1. Checks the forecast using the Open-Meteo API (no API key needed)
  2. Evaluates whether rain is likely (>1mm precipitation)
  3. Only sends a notification if you actually need to know
  4. Otherwise silently does nothing

Delivery mode "none" means the job runs in an isolated session — no messages are sent to your chat unless the agent explicitly decides to send one.

Job 2: Weekly Email Processing

Schedule: Every Monday at 8 AM

{
  "name": "AgentMail Weekly Processing",
  "schedule": {
    "kind": "cron",
    "expr": "0 8 * * 1",
    "tz": "Australia/Melbourne"
  },
  "target": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Check and process new emails from AgentMail inbox."
  },
  "delivery": {
    "mode": "none"
  }
}

This job triggers the email processing script. The key design decisions:

  • Monday morning — before the work week starts, so you have a clean inbox
  • Isolated target — the agent processes emails in a sandbox
  • No delivery — the script sends its own summary email

Job 3: Daily Configuration Backup

Schedule: Every day at 5 PM

{
  "name": "Daily OpenClaw Config Backup",
  "schedule": {
    "kind": "cron",
    "expr": "0 17 * * *",
    "tz": "Australia/Melbourne"
  },
  "target": "isolated",
  "payload": {
    "kind": "agentTurn",
    "message": "Run the OpenClaw configuration backup script."
  },
  "delivery": {
    "mode": "announce"
  }
}

This job runs a shell script that backs up the OpenClaw config files. It uses delivery.mode: "announce", meaning the agent announces the result. If the backup fails, you know immediately.

Why not add this to the website backup? The OpenClaw config contains API keys and tokens — it's backed up separately and stored with tighter permissions (600/700). The website backup covers the public-facing content.

The Patterns

Looking across these jobs, several scheduling patterns emerge:

Pattern 1: Recurring Cron

{ "kind": "cron", "expr": "0 18 * * 5" }

Standard cron expression. Use this for anything that repeats on a regular schedule.

Pattern 2: Conditional ("Only if needed")

This isn't a cron pattern but a job logic pattern. The cron fires every Friday, but the agent decides whether to actually notify you. To the scheduler, the job always runs and reports success. To you, the job only surfaces when it matters.

Delivery Modes

ModeBehaviourUse Case
"announce"Sends agent's response to default channelBackup failures, important notifications
"none"Runs silently, no notificationWeather check (conditional), email processing

Mode "none" is the most interesting for AI agents. It lets the agent decide independently whether to communicate — the cron job is just the trigger.

Lessons Learned

Lesson 1: Timezone Matters

All cron expressions in OpenClaw use the server's timezone unless explicitly overridden with "tz". For a personal agent, specifying "Australia/Melbourne" prevents confusion with UTC-based scheduling.

Lesson 2: Isolated Sessions Prevent Side Effects

Every cron job targets an isolated session. Jobs can't accidentally access or modify other conversations, state is isolated between runs, and errors in one job don't affect others. Even the "daily backup" job, which seems safe, runs isolated. It's a small overhead that prevents big problems.

Lesson 3: Test With Announce First

When setting up a new cron job, start with delivery.mode: "announce" so you can see the agent's output. Once you're confident the logic works, switch to "none" for silent conditional jobs. The weather check was initially set to announce every Friday — after two weeks of "no rain expected" messages, the conditional logic was added.

Lesson 4: Cron Format Is Standard But Fragile

OpenClaw uses standard cron syntax (minute hour day month weekday). The most common mistake is getting the order wrong (it's minute-first, not hour-first). Always verify with a cron tester before relying on a job.

Lesson 5: Separate Sensitive Backups

Config backups containing API keys, tokens, and credentials should be stored separately from public website backups with tighter file permissions (600/700 instead of 644). This limits exposure if the website backup is ever leaked.

What Could Be Better

Dynamic Scheduling

Currently, schedules are static. The agent can't say "remind me about this in 3 days" and have it create a new cron entry. Adding a tool that lets the agent schedule and manage jobs would unlock full autonomous scheduling.

Failure Notifications

Jobs with mode: "none" that fail silently are hard to debug. Adding a health-check that periodically verifies recent cron jobs executed successfully would improve reliability.

Job Dependencies

Some workflows require job sequencing ("after email processing, send a summary"). Currently, each job is independent. A chaining mechanism would enable multi-stage automation.

Final Thoughts

Cron jobs are the unsung heroes of personal automation. Adding an AI agent to the mix turns them from rigid timers into intelligent assistants that check, decide, and act on your behalf.

The key insight is that most cron jobs shouldn't actually do the work — they should trigger the work. The agent brings the intelligence. The cron brings the timing. Together, they automate the predictable parts of your life so you don't have to think about them.

Start with one job. Get it right. Then add another.
