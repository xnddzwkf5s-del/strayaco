External OpenClaw Access Guide

OpenClaw Cloudflare Tunnel Ngrok Tailscale AI

A practical guide to exposing your local OpenClaw dashboard to the internet with HTTPS, based on real-world experience.

Overview

OpenClaw runs on http://localhost:18789 by default. To access it from outside your home network (phone, work, travel), you need a tunnel or reverse proxy with HTTPS. This document covers the approaches we tried, what worked, and lessons learned.

Final working solution: Cloudflare Tunnel + Cloudflare Access (email gate) at https://<your-hostname>.<your-domain>

Approaches Tried

1. Cloudflare Tunnel (Named Tunnel) — ✅ WINNER

Method: Install cloudflared, create a named tunnel, run as a LaunchDaemon, point a DNS CNAME at the tunnel endpoint.

AspectDetail
CostFree
HTTPSYes (Cloudflare edge)
Permanent URLYes
Survives rebootYes (LaunchDaemon)
DNS requiredYes (your own domain)
Auth gateFree via Cloudflare Access (email login)

Setup:

# Install cloudflared
brew install cloudflared

# Authenticate with Cloudflare
cloudflared tunnel login

# Create a named tunnel
cloudflared tunnel create openclaw

# Configure ingress in ~/.cloudflared/config.yml
# tunnel: openclaw
# credentials-file: /Users/you/.cloudflared/openclaw-credentials.json
# ingress:
#   - hostname: <your-hostname>.<your-domain>
#     service: http://localhost:18789
#   - service: http_status:404

# Create DNS record at Cloudflare:
# CNAME <your-hostname> → <tunnel-id>.cfargotunnel.com (proxied)

# Install as system service (auto-starts on boot)
sudo launchctl bootstrap system /Library/LaunchDaemons/com.cloudflare.cloudflared.plist

Pros: Free, permanent URL, full HTTPS, auto-starts on boot, free email auth gate, no open ports on router, works from any browser without client software, works on locked-down company laptops.

Cons: Requires your own domain, requires Cloudflare as DNS provider, several moving parts in setup.

Key lesson: You must use Cloudflare as your DNS provider. The cfargotunnel.com domain is internal to Cloudflare — it cannot resolve publicly without Cloudflare NS. Don't bother trying to use a CNAME at GoDaddy or another registrar.

2. Quick Tunnel (trycloudflare.com) — ❌ FAILED

Method: cloudflared tunnel --url http://localhost:18789 generates a random *.trycloudflare.com URL.

AspectDetail
CostFree
HTTPSYes
Setup time1 minute
ReliabilityPoor

Result: Tunnel connected successfully from our side (cloudflared reported 0 errors, 11 requests proxied), but the trycloudflare.com URL always returned HTTP 404 from Cloudflare's edge. This persisted across multiple tunnel instances with different random subdomains.

Root cause (suspected): trycloudflare.com quick tunnels have a routing issue at Cloudflare's edge. The edge returns 404 before the request reaches the cloudflared client.

3. Ngrok (Free Tier) — ❌ NOT RECOMMENDED

Method: brew install ngrok && ngrok http 18789 generates a random *.ngrok-free.dev URL.

AspectDetail
CostFree tier; $10+/mo for fixed subdomain
HTTPSYes
Setup time2 minutes
ReliabilityGood

Result: Worked immediately. The OpenClaw dashboard loaded correctly once the allowedOrigins config was updated with the exact ngrok URL.

Pros: Fastest setup, no DNS needed, works reliably, good for testing.

Cons: Random URL changes on every restart (free tier), permanent URL costs ~$10/month, need to update config each time the URL changes.

4. Tailscale Funnel — ❌ NOT PRACTICAL

AspectDetail
CostFree (up to 3 users)
HTTPSYes
Setup time5 minutes
Client requirementTailscale client on every accessing device

Result: Requires the Tailscale client on both the host machine AND every accessing device. Impractical for company-managed laptops, shared computers, or guest devices where you can't install software.

Key lesson: For a dashboard you need from anywhere, a browser-based solution like Cloudflare Tunnel is essential. Tailscale Funnel is fine for personal devices you control, but not for accessing from locked-down environments.

OpenClaw Configuration

For any external access method, update your OpenClaw config to allow the external origin:

File: ~/.openclaw/openclaw.json

{
  "gateway": {
    "controlUi": {
      "allowedOrigins": [
        "http://localhost:18789",
        "http://127.0.0.1:18789",
        "https://<your-hostname>.<your-domain>"
      ],
      "allowInsecureAuth": true,
      "dangerouslyDisableDeviceAuth": true
    },
    "auth": {
      "mode": "token",
      "token": "your-gateway-token"
    }
  }
}

Important notes:

  • Wildcards (e.g. *.ngrok-free.dev) do NOT work — use the exact origin URL
  • If you forget to add the URL, the browser will show: "origin not allowed" — the HTML loads but WebSocket auth fails
  • After editing config, run openclaw gateway restart or the old config remains in memory
  • With Ngrok free tier, the URL changes every restart — you must update allowedOrigins and restart each time
  • Use https:// not wss:// in the origins list — OpenClaw handles WebSocket matching internally

Cloudflare Access (Email Gate)

Add an email authentication gate via Cloudflare Access:

  1. Go to Cloudflare Dashboard → Zero Trust
  2. Choose a team name
  3. Go to Access → Applications → Add an application → Self-hosted
  4. Set domain to your tunnel subdomain (e.g. <your-hostname>.<your-domain>)
  5. Add policy: Allow if email ends with you@example.com
  6. Save

This is free on Cloudflare's Zero Trust free plan (up to 50 users).

DNS Migration (GoDaddy → Cloudflare)

  1. Add your domain to Cloudflare (it scans existing DNS records)
  2. Verify all records imported (A, CNAME, MX, TXT for email)
  3. At your registrar, change nameservers to Cloudflare's
  4. Wait 5-30 minutes for propagation
  5. Existing websites and email continue working — Cloudflare auto-proxies traffic

Do NOT add CNAME records at your registrar pointing to cfargotunnel.com — they won't resolve without Cloudflare NS.

Recommendation

Use Cloudflare Tunnel + Cloudflare Access for a permanent, free, secure setup:

  1. Move your domain DNS to Cloudflare (free)
  2. Install and configure cloudflared as a LaunchDaemon
  3. Set up Cloudflare Access for email authentication
  4. Update OpenClaw config with the allowed origin
  5. Done — permanent HTTPS URL that survives reboots

This is the only approach tested that delivers all of: free cost, permanent URL, HTTPS, boot persistence, access control, and works from any browser without client software. Perfect for accessing a headless home server from locked-down work laptops or any device with a web browser.
